# VE 浩鹏素材刷流复核网站需求文档

## 背景

当前 Video Enhancer 链路已经能把每日竞品素材同步到飞书多维表，并通过 `浩鹏接受情况` 沉淀浩鹏反馈。现有复核方式主要依赖多维表和飞书 TopN 卡片：适合批量管理，但不适合连续浏览视频素材，也不够接近业务人员“看到喜欢就点一下”的自然动作。

本需求希望把 VE 多维表中的素材转成一个可浏览、可播放、可快速反馈的网站。第一版服务浩鹏个人复核，不做多人 reviewer 区分，但接口和数据结构预留后续扩展为多人复核。

## 目标

1. 提供一个类似短视频信息流的网页，浩鹏可以连续浏览、播放 VE 素材。
2. 浏览时可以快速给素材反馈，第一版核心动作是“喜欢/点赞”，并能映射到现有 `浩鹏接受情况` 字段。
3. 网站数据来源复用现有 VE 主多维表，不重新发明素材库。
4. 反馈写回现有多维表字段，使浩鹏 TopN、采纳率统计、反馈训练、留存维护等下游链路继续复用同一套反馈事实。
5. 第一版不做多人 reviewer 选择，但 API 层保留 reviewer/status field 参数，后续可以扩展到尉蓝或其他业务人员。

## 非目标

1. 第一版不替代 VE 抓取、分析、同步主流程。
2. 第一版不做复杂推荐算法，只按明确排序规则返回素材。
3. 第一版不做公开视频站或外部分享站，只面向内部使用。
4. 第一版不新增一套独立审核结论体系；点赞和反馈必须最终落到现有多维表记录。
5. 第一版不实现多人冲突处理、权限分层、评论线程、素材下载管理。

## 用户与使用场景

### 主要用户

浩鹏。

### 核心场景

1. 浩鹏打开网站，默认看到最近一天或指定日期的 VE 素材。
2. 页面进入沉浸式刷流：一个屏幕聚焦一个素材，视频可播放，图片素材可展示封面/预览。
3. 浩鹏看到有价值素材时点“喜欢”。
4. 系统立即写回多维表，把该素材的 `浩鹏接受情况` 更新为 `采纳`。
5. 浩鹏也可以使用更多反馈动作：`入素材库`、`不采纳`、`重复抓取`、`待定`。
6. 已反馈素材在页面上显示当前状态，避免重复判断。

## MVP 功能范围

### 素材流

- 默认读取 `VIDEO_ENHANCER_BITABLE_URL` 指向的 VE 主表。
- 默认筛选 `抓取日期 = 最近有素材的一天`，也支持通过日期参数切换。
- 每条素材至少展示：
  - 视频或封面预览
  - 产品
  - 平台
  - 标题或核心卖点
  - 玩法
  - Hook
  - 风险等级
  - 当前 `浩鹏接受情况`
- 支持上下切换素材，桌面端可用键盘方向键，移动端可滑动。
- 当前素材视频默认可播放；切换到下一条时暂停上一条。

### 反馈动作

第一版提供以下动作：

| 页面动作 | 写回字段 | 写回值 | 用途 |
| --- | --- | --- | --- |
| 喜欢 | `浩鹏接受情况` | `采纳` | 最快表达“这个值得做/值得采纳” |
| 入库 | `浩鹏接受情况` | `入素材库` | 有价值但更适合作为素材库积累 |
| 不喜欢 | `浩鹏接受情况` | `不采纳` | 不值得继续跟进 |
| 重复 | `浩鹏接受情况` | `重复抓取` | 明确是重复素材 |
| 待定 | `浩鹏接受情况` | `待定` | 暂时不判断或需要后续确认 |

反馈写回成功后，页面必须立刻更新本地状态，并给出轻量成功提示。写回失败时不得假装成功，应保留原状态并显示可重试提示。

### 列表与筛选

MVP 至少支持：

- 日期筛选：今天、昨天、指定日期。
- 状态筛选：全部、未反馈、已喜欢、已入库、不采纳、重复、待定。
- 产品筛选：按产品名称过滤。
- 排序：
  - 默认按现有表记录顺序或创建时间倒序。
  - 可选优先展示未反馈素材。

### 详情信息

每条素材允许打开详情面板，展示多维表中更完整的分析字段：

- 广告 ID
- 视频链接
- 封面图链接
- 正文/标题
- 核心卖点
- Hook解析
- 脚本/口播
- 玩法资产/玩法变种
- 玩法指纹
- 模板指纹
- 差异点
- 素材标签
- 日内相似素材数

详情面板还应提供“打开多维表记录或主表”的入口。若无法定位单条记录，则至少提供主多维表入口。

## 数据设计

### 数据源

第一版以飞书多维表为在线事实源：

- 环境变量：`VIDEO_ENHANCER_BITABLE_URL`
- 认证：项目 `.env` 中的 `FEISHU_APP_ID` / `FEISHU_APP_SECRET`
- 默认反馈字段：`浩鹏接受情况`

可选优化：服务端可以使用短时缓存减少多维表读取频率，但写回必须落到多维表。

### 素材响应模型

服务端返回给前端的素材对象使用稳定内部字段名，避免前端直接耦合中文表字段：

```json
{
  "record_id": "recxxxx",
  "ad_key": "creative-id",
  "crawl_date": "2026-06-08",
  "product": "PixVerse",
  "platform": "tiktok",
  "title": "...",
  "core": "...",
  "hook": "...",
  "play_label": "...",
  "risk_level": "低风险",
  "video_url": "https://...",
  "cover_url": "https://...",
  "preview_img_url": "https://...",
  "status": "待定",
  "reviewer": "haopeng",
  "source_fields": {
    "template_fingerprint": "...",
    "material_tags": "..."
  }
}
```

### 反馈写回模型

前端提交：

```json
{
  "record_id": "recxxxx",
  "status": "采纳",
  "reviewer": "haopeng"
}
```

第一版服务端固定把 `reviewer=haopeng` 映射到 `浩鹏接受情况`。后续多人扩展时，可以通过配置表扩展：

```json
{
  "haopeng": "浩鹏接受情况",
  "weilan": "尉蓝接受情况"
}
```

## 技术设计

### 推荐落地形态

在当前 `ua素材` 仓库内新增一个轻量内部 Web 服务：

- 后端：Python，复用现有飞书多维表解析、tenant token、字段归一化代码。
- 前端：单页应用，提供短视频刷流体验。
- 部署：先支持本地/内网运行；后续可部署到 Mac mini 或内部站点。

这样可以最大化复用现有 `.env`、VE 字段口径和 Python 代码，避免单独维护一套 Node 服务凭证。

### 后端模块建议

新增模块建议放在：

- `ua_workflows/video_enhancer/review_site/`

建议拆分：

- `bitable.py`：读取 VE 主表、字段归一化、写回反馈。
- `models.py`：素材对象和反馈请求模型。
- `server.py`：HTTP API。
- `static/` 或前端独立目录：刷流 UI。

API 建议：

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `GET` | `/api/materials` | 拉取素材流，支持 date/status/product/cursor/limit |
| `POST` | `/api/materials/{record_id}/feedback` | 写回反馈状态 |
| `GET` | `/api/options` | 返回可选日期、产品、状态、默认 reviewer |
| `GET` | `/healthz` | 健康检查 |

### 前端交互

第一屏即素材刷流，不做营销式首页。

布局：

- 主区域：视频/图片预览，占据主要屏幕。
- 右侧或底部：反馈按钮。
- 顶部：日期、状态、产品筛选。
- 底部或抽屉：素材标题、产品、玩法、Hook。
- 详情按钮：打开完整字段面板。

快捷键：

- `ArrowDown` / `j`：下一条
- `ArrowUp` / `k`：上一条
- `l`：喜欢，写回 `采纳`
- `i`：入素材库
- `x`：不采纳
- `r`：重复抓取
- `u`：待定

快捷键必须只作为增强能力，页面按钮仍是主路径。

## 状态与错误处理

1. 多维表读取失败：显示错误页，说明无法读取素材，并提示检查 `.env` 和飞书权限。
2. 素材无视频：退化展示封面图或预览图。
3. 视频无法播放：显示封面和“打开原链接”按钮。
4. 写回失败：保留原状态，展示错误提示，允许重试。
5. 多端同时改同一条：MVP 允许最后写入覆盖；写回后以前端最新响应为准。
6. 字段不存在：服务端启动或首次请求时返回明确错误，指出缺失字段名。

## 安全与权限

MVP 运行在内部环境，不做公网开放。

最低要求：

- 服务端不把 `FEISHU_APP_SECRET` 暴露给浏览器。
- 所有飞书 API 调用只在服务端进行。
- 前端只拿到渲染素材所需字段，不暴露完整原始记录。
- 部署时通过内网、VPN、反向代理或基础访问控制限制访问。

后续如果要公网访问，必须补充登录鉴权、操作审计和 CSRF/写接口保护。

## 验收标准

1. 打开网站后，可以看到 VE 主表中指定日期的素材流。
2. 至少 20 条素材可以连续切换浏览，视频或封面能正常展示。
3. 点击“喜欢”后，多维表对应记录的 `浩鹏接受情况` 更新为 `采纳`。
4. `入库 / 不喜欢 / 重复 / 待定` 均能写回正确值。
5. 页面刷新后，已写回状态仍能从多维表读出。
6. 日期、状态、产品筛选可用。
7. 当飞书凭证或字段缺失时，错误信息明确，不出现静默失败。
8. 不影响现有 VE 抓取、同步、浩鹏 TopN、反馈训练脚本。

## 测试计划

### 单元测试

- 多维表字段归一化：中文字段转内部素材对象。
- 状态映射：页面动作转 `浩鹏接受情况` 值。
- reviewer 映射：`haopeng -> 浩鹏接受情况`，未知 reviewer 返回错误或默认拒绝。
- 筛选参数：date/status/product/cursor/limit。

### 集成测试

- 使用 mock 飞书 API 验证读取素材列表。
- 使用 mock 飞书 API 验证 batch update payload。
- 验证写回失败时 API 返回错误，前端不更新为成功。

### 浏览器测试

- 打开素材流页面。
- 切换下一条/上一条。
- 点击喜欢并观察状态变化。
- 打开详情面板。
- 在无视频素材上退化展示封面。

## 里程碑

### Milestone 1：只读刷流

- 后端能从 VE 主表读取素材。
- 前端能按日期浏览素材。
- 视频/封面展示可用。

### Milestone 2：反馈写回

- 点赞写回 `浩鹏接受情况=采纳`。
- 完整反馈动作写回。
- 页面状态与多维表一致。

### Milestone 3：可用性补齐

- 筛选、快捷键、详情面板。
- 错误提示和加载状态。
- 基础浏览器验证。

### Milestone 4：部署与内测

- 提供启动脚本。
- 在内部机器上跑起来。
- 浩鹏试用一到两天，根据真实反馈调整按钮、排序和字段展示。

## 后续扩展

1. 多 reviewer：增加 reviewer 选择，按配置写不同字段。
2. 推荐排序：接入浩鹏 TopN 分数，优先展示模型认为更可能采纳的素材。
3. 操作日志：单独记录谁在何时把哪条素材改成什么状态。
4. 评论：支持浩鹏写一句原因，后续同步到备注或独立字段。
5. 离线缓存：每日同步后生成站点缓存，提高浏览速度。
6. 移动端体验增强：适配手机全屏滑动和触控反馈。
7. 训练闭环增强：把网站反馈直接作为反馈训练链路的一等输入。
